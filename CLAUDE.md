# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build Commands

```bash
# Configure (Ninja recommended)
cmake -B build -G Ninja

# Build
cmake --build build

# With explicit compiler (Clang)
cmake -B build -G Ninja -DCMAKE_C_COMPILER=clang -DCMAKE_CXX_COMPILER=clang++
cmake --build build

# Visual Studio
cmake -B build -G "Visual Studio 17 2022" -A x64
cmake --build build --config Debug
```

C++23 is required. Exceptions are disabled (`ENGINE_NO_EXCEPTIONS=ON`). Output lands in `build/{CompilerName}{PlatformName}/`.

## Running Tests

Tests run automatically during `CoreManager::OnApplicationPreInit()`. They are enabled/disabled per-class via `config/TestsConfig.xml`. There is no standalone test binary — tests run as part of engine startup.

## Architecture Overview

BagiEngine is a C++23 game engine with modular ECS architecture (EnTT). The main entry is `src/main.cpp` → `Application::Initialize()/Run()`.

**Module dependency order (bottom → top):**
```
BECore → Math → Events → TaskSystem → SDL → Application
```

**Key subsystems in `src/Modules/BECore/`:**

| Subsystem | Purpose |
|-----------|---------|
| `GameManager/CoreManager` | Central service locator — owns FileSystem, ConfigManager, Logger, SceneManager, ResourceManager, etc. |
| `Scene/` | Scene graph (SceneNode → IComponent), serialization |
| `Reflection/` | IPropertyVisitor, ISerializer/IDeserializer, BE_CLASS codegen markers |
| `Resource/` | TextureLibrary, resource pooling |
| `Renderer/` | Multi-backend renderer (SDL3 default, Vulkan optional) |
| `Events/` | CRTP-based pub/sub (`EventBase<T>`, `SubscriptionHolder` for RAII) |
| `TaskSystem/` | Async coroutines |

UI widgets live in `src/Widgets/`. `SceneEditorWidget` handles orchestration/UI flow; `ComponentInspectors` handles per-field editing via `IPropertyVisitor`. Scene editor behavior is driven by `config/SceneConfig.xml`.

Reflection code is generated by `CI/meta_generator/meta_generator.py` (Python, regex-based). `BE_CLASS` and `BE_REFLECT_FIELD` macros are the generation triggers.

## Code Style

**Naming:** `PascalCase` for classes/methods, `camelCase` for locals, `_camelCase` for private members, `PascalCase` for namespaces.

**DO:**
- Use EASTL containers over `std::` (esp. `eastl::vector`, `eastl::hash_map`, `eastl::string`)
- Use `PoolString` for identifier/key strings; use `"Key"_intern` for string literals
- Use `IntrusivePtr<T>` (never raw `new`/`delete`)
- Use `CORE_ENUM(Name, Type, Values...)` instead of `enum class`
- Use `ASSERT()` (not `std::assert`), `FATALERROR()` for critical failures
- Use `std::expected<T, E>` for recoverable errors (no exceptions)
- Add `override` for virtual methods, `#pragma once`, namespace closing comments

**DON'T:**
- Use `std::string` where `PoolString` fits
- Use `enum class` — use `CORE_ENUM`
- Use C-style casts

## Serialization / Reflection

Use split interfaces: `ISerializer` (write) and `IDeserializer` (read). Do not introduce archive-style APIs. Codegen via `CI/meta_generator/` is the source of truth.

## After Making Changes

- Apply `clang-format` to changed files before pushing.
- If includes were added or modified, run `strip-pch-includes` to avoid duplicating PCH-provided headers.
- For complex tasks, save a plan to `.cursor/plans/PLAN_NAME.md` and update it as work progresses.
